# 开源计划（OPEN_SOURCE）｜探店复盘 App（暂定名）

目标：当 MVP 跑通并稳定后，将项目逐步开源，方便大众自建与贡献。

## 1. 建议的开源时机

满足以下条件再开源更稳：
- 你们自己连续使用 4~8 周，核心流程稳定
- 已明确“默认私密/公开发布”的治理策略
- 已完成最小的密钥与配置隔离（不会把生产 key 带进仓库）

## 2. 开源范围建议（分阶段）

### 阶段 A（推荐）
- 开源：数据模型（migration）、API 规范、客户端代码
- 提供：一键部署模板（例如 Supabase schema / Docker compose）
- 不开源：生产环境配置、内部运营数据、任何密钥

### 阶段 B
- 增加：最简 Admin 管理端（举报/反馈处理）
- 增加：圈子协作（若做）

## 3. 仓库结构建议

单仓（monorepo）示例：
- `apps/mobile/`（Flutter 或 RN）
- `apps/admin/`（可选）
- `services/api/`（自建后端时）
- `packages/shared/`（DTO/工具）
- `docs/`（本套 md 文档）
- `infra/`（部署脚本/模板）

## 4. 许可证（License）建议

常见选择：
- **MIT**：最宽松，利于传播（推荐）
- **Apache-2.0**：对专利条款更明确

> 若你担心被拿去直接商业化且不回馈，可考虑更强约束的许可证，但会降低传播与贡献。

## 5. 安全与密钥（必须）

- 所有密钥只允许出现在：
  - `.env`（不提交）
  - CI/CD Secret
  - 部署平台的 Secret 管理
- 仓库提供：
  - `.env.example`（示例，不含敏感值）
  - 安全检查（如 secret scan，或提交前 hook）

## 6. 贡献规范（CONTRIBUTING 摘要）

建议在仓库中补充：
- 如何本地运行（mobile/api/admin）
- 提交规范（Conventional Commits）
- PR 模板（背景、改动、验证）
- Issue 模板（bug/feature request）

## 7. 内容与合规声明（公开发布相关）

开源 README 建议包含：
- 项目定位：个人体验分享工具
- 使用者自负：部署者对其站点内容负责
- 参考治理：提供举报/下架/封禁机制，但不保证自动审核
