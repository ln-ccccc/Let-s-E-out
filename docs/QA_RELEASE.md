# 测试与发布（QA_RELEASE）｜探店复盘 App（暂定名）

## 1. 测试目标

- 核心链路稳定：登录 → 新建复盘 → 列表/筛选 → 公开 → 收藏 → 举报/反馈
- 关键数据不丢：复盘内容、公开状态、收藏状态、举报/反馈提交回执
- 弱网/断网可用性：至少保证“错误提示明确、可重试、不误删”

## 2. 测试分层

### 2.1 单元测试（建议）
- Model/DTO 映射
- 业务规则（必填校验、权限错误映射）

### 2.2 集成测试（建议）
- API client：错误码、重试、token 过期处理
- 关键接口契约：创建复盘、更新、公开列表、收藏、举报、反馈

### 2.3 手工验收清单（MVP 必须）
1) 登录成功后可进入首页  
2) 3 分钟内完成一条复盘（必填项校验正常）  
3) 复盘在“我的列表”可见，详情展示正确  
4) 筛选条件生效（至少：会不会再去、时段、标签任一）  
5) 私密复盘不可在公开页出现  
6) 公开复盘可被其他账号看到并收藏  
7) 举报提交成功，有回执；管理侧可查到  
8) 反馈入口在「我的」一级可见；提交成功，有回执  
9) 断网时：提交动作提示失败，不会造成“假成功”  

## 3. 版本与发布策略

### 3.1 版本号（SemVer）
- `MAJOR.MINOR.PATCH`
- MVP 建议从 `0.1.0` 开始

### 3.2 发布渠道
- iOS：TestFlight（内测）→ App Store（公测/正式）
- Android：内测包（自分发/应用市场）→ 正式上架

### 3.3 发布节奏（建议）
- MVP 阶段：每周 1-2 次内测发布
- 稳定后：两周一个 minor 版本

## 4. CI（持续集成）建议

至少包含：
- Lint/Format（强制）
- Build（iOS/Android）
- 单元测试（若有）

发布（可选）：
- 自动打包内测产物
- 自动生成 changelog（从 Conventional Commits）

## 5. 数据与迁移

如果使用 Postgres：
- 任何 schema 变更必须有迁移脚本（Supabase migration 或自建 migration）
- 迁移需可回滚（至少保证不会把线上数据写坏）

## 6. 回滚与事故处理（最简）

当线上出现严重问题：
- 立即下线/禁用某功能开关（若有 feature flag）
- 回滚到上一稳定版本（客户端可退回、后端可切换版本）
- 记录事故：时间线、影响范围、根因、修复、预防措施
