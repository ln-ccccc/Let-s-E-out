# 运营与反馈（OPS）｜探店复盘 App（暂定名）

本文件描述 MVP 阶段必须具备的“反馈可达”与“公开内容治理”最小闭环。

## 1. 反馈渠道（体验反馈）设计

### 1.1 入口位置（必须容易找到）
- 「我的」页：一级入口 **反馈与建议**
- 复盘详情页：二级入口（内容相关问题/产品建议的快捷入口）
- 异常页/空状态页：提示“点此反馈”

### 1.2 反馈表单字段（MVP）
必填：
- `type`：bug/建议/内容/其他
- `message`：问题描述（建议引导：复现步骤 + 期望结果）

可选：
- `screenshot`：截图（0~3）
- `contact`：联系方式（邮箱/微信号等，明确“可不填”）
- `appVersion`、`os`、`device`：自动采集（若允许）

### 1.3 回执与状态流转
用户提交后必须看到：
- “已收到”提示
- 反馈单号（可选）

后台状态（最简）：
- `received` → `in_progress` → `done`

> 是否给用户“处理结果回复”：MVP 可不做消息通知，只保留内部字段 `reply` 以便后续扩展。

## 2. 内容治理（公开发布）最小闭环

### 2.1 为什么需要最小治理
公开发布会涉及：
- 辱骂/攻击性语言
- 诽谤、商家纠纷
- 隐私泄露（拍到人脸、手机号、车牌等）

MVP 目标不是做“强审核”，而是：
**可举报 → 可处理 → 可下架/封禁 → 可追溯**

### 2.2 举报入口（MVP）
位置：
- 公开复盘详情页：**举报**

举报原因（枚举）：
- 辱骂/攻击
- 造谣诽谤/引战
- 色情低俗
- 隐私泄露
- 其他

### 2.3 管理端处理动作（MVP）
管理员可执行：
- 标记举报状态：open/triaged/resolved/rejected
- **下架复盘**：将 `publish_status` 改为 `private` 或增加 `takedown=true`
- **封禁用户**：`users.status = banned`
- 记录处理信息：handled_by / handled_at / resolution

### 2.4 建议的“默认策略”（降低风险）
- 复盘默认私密，用户手动公开（MVP 推荐）
- 首次公开可提示“发布规范与免责声明”

## 3. 发布规范（给用户看的简短规则）

建议在“公开发布”前展示/勾选：
- 不发布辱骂/攻击性内容
- 不发布未经核实的严重指控
- 不泄露他人隐私
- 这是个人体验分享，观点不代表事实

## 4. 运营数据（MVP 关注）

- 反馈提交量（按类型）
- 平均处理时长
- 举报量、举报原因分布
- 下架率、封禁数
- 公开发布占比（私密→公开转化）

## 5. 客服与联系（可选）

若你希望对外公开联系方式：
- 在「关于」页放一个邮箱/表单链接
- 同时在开源 README 中放“问题反馈”渠道（GitHub Issues / Discussions）
