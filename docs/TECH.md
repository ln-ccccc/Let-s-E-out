# 技术方案（TECH）｜探店复盘 App（暂定名）

## 1. 技术目标

- **快速迭代**：优先保障 MVP 速度与可维护性
- **跨平台一致**：iOS/Android 统一一套代码（建议）
- **可开源/可自建**：核心后端与数据结构可迁移/可替换
- **最小内容治理**：公开发布需要举报、风控与基础管理能力
- **体验简洁**：不追求花哨视觉，信息密度与可读性优先

## 2. 推荐技术栈（默认建议）

> 若你已偏好某栈（Flutter / React Native / 原生），可替换，本文件保持“接口与数据模型稳定”。

### 2.1 客户端
- **Flutter**（推荐）
  - 优点：跨端一致、UI 实现效率高、工程化成熟
  - 缺点：需要适应 Dart 生态
- 备选：React Native（若团队更熟 JS/TS）

### 2.2 后端（MVP 推荐托管，兼顾自建）
两条路径：

**路径 A：Supabase（推荐）**
- Postgres + Auth + Storage + Row Level Security
- 很适合：MVP、快速搭建、权限控制（RLS）清晰

**路径 B：自建 API**
- NestJS（Node/TS）或 FastAPI（Python）+ Postgres
- 适合：强定制、后期规模化、完全自托管

> 文档默认以“REST API + JWT/Session”的抽象描述，不绑死具体实现。

### 2.3 存储
- 图片：对象存储（Supabase Storage / S3 兼容）
- 文本与结构化数据：Postgres

### 2.4 观测与反馈
- 崩溃与性能：Sentry（可选）
- 反馈工单：先用自建表（Feedback）+ 管理端（最简）即可

## 3. 系统架构概览

### 3.1 模块划分
- Mobile App
  - Auth：登录/会话
  - Visit：复盘创建/编辑/详情
  - Place：店铺实体（可先弱化）
  - Public：公开浏览/收藏
  - Search：搜索筛选
  - Safety：举报入口
  - Feedback：产品反馈入口（全局可达）
- Backend API / BaaS
  - Auth & 用户资料
  - Places / Visits / Favorites
  - Reports（举报）
  - Feedbacks（反馈）
  - Admin（最简管理接口或控制台）

### 3.2 权限模型（原则）
- 私密复盘：仅作者可读写
- 公开复盘：所有人可读；仅作者可改；管理员可下架/封禁
- 收藏：仅自己可见
- 举报：任何登录用户可提交；仅管理员可看全量
- 反馈：登录用户可提交；仅管理员可看全量

## 4. 关键技术决策

### 4.1 “店铺 Place”与“复盘 Visit”分离（必须）
原因：
- 同一店铺可多次到店，多条复盘不互相覆盖
- 便于按“时间段/场景”积累经验

### 4.2 搜索与筛选策略（MVP）
MVP 不做全文搜索服务：
- 店名：ILIKE/前缀匹配
- 标签/场景/时段：枚举与数组字段
- 区域：先用文本（武汉：行政区/商圈），后续再接 POI/经纬度

### 4.3 内容治理最小闭环
MVP 至少包含：
- 举报提交（Report）
- 管理端查看（列表 + 详情）
- 处理动作：标记已处理 / 下架内容 / 封禁用户（最简）

## 5. 客户端工程结构建议（示例）

> 以 Flutter 为例；RN 类似。

- `lib/`
  - `app/` 路由、主题、全局依赖注入
  - `features/`
    - `auth/`
    - `visits/`
    - `places/`
    - `public/`
    - `search/`
    - `favorites/`
    - `reports/`
    - `feedback/`
  - `shared/`
    - `network/` API client、拦截器、错误处理
    - `models/` DTO/实体
    - `ui/` 通用组件（按钮、卡片、表单项）
    - `utils/` 时间、校验、埋点封装等

## 6. 非功能需求（NFR）

### 6.1 性能与体验
- 列表首屏 < 1.5s（弱网另议）
- 表单保存：本地乐观更新 + 后台同步（可选）

### 6.2 可靠性
- 网络错误统一提示 + 可重试
- 提交类动作（复盘/举报/反馈）必须有“成功回执”

### 6.3 安全与隐私
- 默认私密发布（MVP 建议）
- 图片 EXIF（定位信息）可选清理（可在 V1 做）
- 管理接口必须鉴权，操作留审计日志（最简：谁在何时做了什么）

## 7. 管理端（MVP 最简实现）

可选实现方式：
- Supabase Dashboard（最快）
- 一个最简 Web Admin（仅内部使用）

必须能力：
- 查看反馈列表、状态流转
- 查看举报列表、处理内容、封禁用户

## 8. 交付物与里程碑（建议）

### MVP 交付物
- App：登录、复盘、搜索筛选、公开发布、收藏、举报、反馈
- 后端：对应 API/表结构 + 权限策略
- 管理端：举报/反馈处理最小闭环

### 里程碑（业余开发参考）
- W1：原型 + 表结构 + API 定稿
- W2-3：MVP 功能开发
- W4：治理/反馈闭环 + 内测
